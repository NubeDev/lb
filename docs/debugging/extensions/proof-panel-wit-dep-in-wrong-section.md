# wasm guests unbuildable — `lb-sdk` was a build-dep, so `DEP_LB_SDK_WIT` was never set

- **Date:** 2026-07-14
- **Area:** extensions (Tier-1 wasm guest build ← the SDK/WIT out-of-tree split)
- **Status:** fixed
- **Found by:** full-suite triage — chasing `proof_panel_test` "17/23 known-red" instead of accepting it
- **Blast radius:** TWO copies of one line. `extensions/proof-panel/Cargo.toml` (killed
  `proof_panel_test`) **and `crates/devkit/templates/wasm/Cargo.toml.tmpl` — the scaffold template,
  so every extension anyone generated with `devkit scaffold` was born unbuildable** (killed
  `build_test` + `devkit_e2e_test`). Three "known-red" binaries, 20 failing tests, one root cause.

## Symptom

`proof_panel_test` failed 17 of 23 with a missing artifact, NOT an assertion:

```
missing component at …/extensions/proof-panel/target/wasm32-wasip2/release/proof_panel_ext.wasm
  (No such file or directory (os error 2)). Build it first: …
```

Building it by hand then failed in the guest's own build script:

```
thread 'main' panicked at build.rs:12:10:
DEP_LB_SDK_WIT — lb-sdk must be a dependency exporting its WIT via `links`: NotPresent
```

## Root cause — one line, in the wrong manifest section

`proof-panel` declared the SDK under **`[build-dependencies]`**:

```toml
# The authoritative WIT lives in the standalone SDK; a build-time dep only — …
[build-dependencies]
lb-sdk = { git = "…/lb-ext-sdk", tag = "sdk-v0.2.1" }
```

Cargo passes `DEP_<links>_<key>` metadata to a package's build script **only for that package's
normal `[dependencies]`** — never for its `[build-dependencies]`. So `DEP_LB_SDK_WIT` was unset and
`build.rs` panicked before `wit_bindgen::generate!` could ever run.

The comment shows the reasoning that produced it: *"a build-time dep only"*. Intuitive — the WIT is
only needed at build time, so surely it is a build-dependency — and exactly backwards. `links`
metadata flows to the **dependent's** build script, so the crate exporting it must be a normal dep of
the package whose `build.rs` reads it.

The proof was already in-tree: `extensions/hello` has a **byte-identical `build.rs`**, the same
`lb-sdk` tag, and builds fine — because it declares `lb-sdk` under `[dependencies]`, with a comment
spelling the rule out:

> *"A normal dep (not build-dep) because cargo only hands `DEP_*` metadata to a dependent whose build
> script's package links it via `[dependencies]`."*

Two extensions, one rule, one of them written the other way. Introduced when the WIT moved out to the
standalone `lb-ext-sdk` repo (extensions were previously resolving a sibling `../../sdk` path).

## The same line, in the scaffold template — the worse copy

`crates/devkit/templates/wasm/Cargo.toml.tmpl` carried the **identical** `[build-dependencies]`
stanza and the identical "a build-time dep only" comment — one authoring pass clearly produced both
files during the SDK split. Its `build.rs.tmpl` reads `DEP_LB_SDK_WIT` at the same `build.rs:12`.

So **every wasm extension generated by `devkit scaffold` was born unbuildable** — the devkit's whole
promise (scaffold → build → publish → call) broken at step 2, for every user, since the split. It
surfaced as two more "known-red" binaries whose panics point at the same `build.rs:12:10`:

- `lb-devkit build_test::builds_generated_wasm_with_real_cargo_when_target_is_available`
- `lb-host devkit_e2e_test::scaffold_build_publish_wasm_then_call_tool`

## Fix

Move `lb-sdk` from `[build-dependencies]` to `[dependencies]` in **both** copies
(`extensions/proof-panel/Cargo.toml` and `crates/devkit/templates/wasm/Cargo.toml.tmpl`), mirroring
`hello`, and replace the misleading comment with the rule + why it is not a build-dep.

## Verification

| | before | after |
|---|---|---|
| `extensions/proof-panel/build.sh` | panics (`DEP_LB_SDK_WIT … NotPresent`) | builds `proof_panel_ext.wasm` (220 KB) + `remoteEntry.js` |
| `cargo test -p lb-host --test proof_panel_test` | **17 failed / 6 passed** | **23 passed / 0 failed** |
| `cargo test -p lb-devkit --test build_test` | **1 failed / 1 passed** | **2 passed / 0 failed** |
| `cargo test -p lb-host --test devkit_e2e_test` | **1 failed / 1 passed** | **2 passed / 0 failed** |

The `devkit_e2e_test` pass is the one that matters: it scaffolds a fresh extension, builds it,
publishes it, and calls its tool — the real proof that a generated guest works end to end again.

## Lesson

**Three binaries (20 tests) sat on the known-red list for weeks as "code failures". They were one
line, duplicated once.** The failing tests never asserted anything about the code under test — they
died in a fixture, on a missing file, or in a build script.

**A bug in a scaffold template is not one bug — it is one per user, on their first minute.** The
`build_test` / `devkit_e2e_test` reds *were* the alarm for exactly that, correctly firing since the
SDK split; being on the accepted list is what kept anyone from reading them. When a template test
goes red, it is reporting on everyone's future project, not just on itself.

That is the real cost of a known-red list: it is a place where a cheap fix can hide indefinitely,
because "expected to fail" stops anyone from reading the failure. **Read the failure, not the list.**
A test that dies on a *missing artifact* is not testing the code and is not evidence of anything —
triage it by its message (`missing …`, `No such file`, `run: cargo build -p …`), which names a build
prerequisite, not a defect.

Corollary for `links` crates: if a build script reads `DEP_FOO_BAR`, the crate exporting `links =
"foo"` must be a **normal** dependency of that same package. "It is only needed at build time" is the
wrong instinct — check a working sibling before inventing the section.
