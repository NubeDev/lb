# Debugging — working history

The project's debugging memory: every issue and how it became working, so nothing is
debugged twice. **Append-only and symptom-led.**

- How this works and the entry template: `../scope/debugging/debugging-scope.md`.
- One file per issue, named by the symptom: `<area>/<symptom-slug>.md`.
- Add a row below when you open an entry; update its status when it closes.

| 2026-06-30 | flows | the canvas "is this flow running?" banner read the **dormant** `flow.cron`/`next_attempt_ts` (no writer since triggers moved to per-node cursors), so an armed cron flow showed as **idle after a restart/reload**; and a headless (cron/source) flow had **no Stop button** at all (the existing Stop only renders for a non-terminal live run, which a finite-firing headless flow never has between firings). Fixed (UI only) by deriving the armed/enabled truth from the authoritative durable `flows.node_state` (+ graph-derived "scheduled" so it holds when disabled), and adding a durable **Deploy/Stop** toggle bound to `flows.enable` — the headless lifecycle switch, correct after restart | resolved | [flows/armed-banner-reads-dormant-cron-no-stop-for-headless.md](flows/armed-banner-reads-dormant-cron-no-stop-for-headless.md) |
| 2026-06-30 | flows | a flow could hold only **ONE trigger** — `flows.save` rejected a second cron ("a flow has one schedule"), and the same wall blocked two MQTT subs / webhooks; separately, one run fired the **whole graph** (every trigger at once), not "this trigger down its own wires". Root cause (design): reactive state (`flow.cron`/`flow.next_attempt_ts`) was hoisted to the **flow**, and `create_run` enqueued every indegree-0 node. Fixed by moving the cursor to a per-node `flow_trigger_state` (the reactor scans **trigger nodes**, N independent), per-wire **subgraph** runs (`reachable_from`/`indegrees_within` + `create_run{entry}`), and dropping the single-schedule rejection. Also shipped a real stateful `counter` (durable `flow_node_memory` + atomic `lb_store::increment`) for the "count goes up" ask | resolved | [flows/flow-level-cron-rejects-multiple-triggers.md](flows/flow-level-cron-rejects-multiple-triggers.md) |
| 2026-06-30 | flows | opening an armed flow showed a **frozen last-run "DONE" snapshot** + a contradictory "54 runs / no runs yet" banner; "the count isn't going up". Root cause (design): the canvas's only runtime view was a finite `flows.runs.get` snapshot, but the spec's **persistent per-node state** (Decision 5 `flow_node_state`, last-value updated in place each scan — already WRITTEN by `record_outcome`) had **no read verb** and was never painted. Fixed by adding `flows.node_state` (+ `GET /flows/{id}/node_state`), painting it as the canvas BASE steady-state with the run snapshot overlaid, an honest armed banner, and deterministic multi-trigger cron derivation. Live: node-state rev advanced 59→60 on a cron firing (updated in place) | resolved | [flows/canvas-shows-finite-run-not-persistent-node-state.md](flows/canvas-shows-finite-run-not-persistent-node-state.md) |
| 2026-06-30 | flows | a `mode:cron` flow trigger **never fired** ("added a cron trigger every min, count never goes up") — two disconnected gaps: (1) `react_to_flows_cron`/`reconcile_flows` were only ever called from tests — **no production loop ticked them**; (2) the canvas writes the schedule to the trigger node's `config.cron` but the reactor scans the top-level `flow.cron`, and `flows.save` never derived one from the other. Fixed by `spawn_flow_reactors` (a detached per-node tick over the configured ws, live clock) wired into node boot + `derive_cron_from_trigger` at save + switching the node binary to `Gateway::new_live`. Live: a `* * * * *` trigger now fires every minute on its own, runs settle with real values | resolved | [flows/cron-trigger-never-fires-no-reactor-driver.md](flows/cron-trigger-never-fires-no-reactor-driver.md) |
| 2026-06-30 | flows | the run id was **constant for the node's whole uptime** — `gw.now` was frozen at gateway construction and the manual run id was `"{flow_id}-run-{now}"`, so every `flows.run` for a flow resolved to one id. That re-drove the same terminal run (churn + flickering Stop/Resume + "no values") AND made overlapping runs hammer the same records (the rev race below). Live: 8 concurrent runs returned the SAME id + a wall of conflicts. Fixed by minting a ULID per manual run (caller-supplied id still honored) + unfreezing `gw.now` to a live accessor (fixed-clock test seam kept) | resolved | [flows/frozen-gw-now-collides-run-ids.md](flows/frozen-gw-now-collides-run-ids.md) |
| 2026-06-30 | flows | run-store / `lb-jobs` writes raced the store's monotonic `rev` RMW under concurrent same-record writers → `Invalid revision '174'` / `read or write conflict … can be retried` (made certain by the constant run id above). `lb_store::write` is atomic per write but not isolated across two writers of one `table:id`. Fixed with a store-level `write_locked` (per-`(ws,table,id)` async lock + bounded retry-on-conflict, the `capped_insert` shape) adopted by run_store + jobs, plus a create-if-absent idempotent seed | resolved | [flows/run-store-rev-conflict-under-concurrency.md](flows/run-store-rev-conflict-under-concurrency.md) |
| 2026-06-30 | flows | backgrounding the manual run (`tokio::spawn(coordinator::drive)`) failed to compile — "future cannot be sent between threads safely", the compiler pointing at `flows_run_async` itself. Not a real non-`Send` capture (a standalone `assert_send(drive(...))` passed) but an **async opaque-type recursion**: a `tool` node can call `flows.run`, so `flows_run_async → drive → tool → dispatch → flows_run_async` is an infinitely-recursive `impl Future` the compiler can't size or prove `Send`; boxing the `.await`s didn't help (the cycle is in the type, not the indirection). Fixed by type-erasing the recursion edge to a **concrete** `call_flows_tool_boxed` (`Pin<Box<dyn Future + Send>>`) at the dispatch boundary | resolved | [flows/async-run-not-send-recursion.md](flows/async-run-not-send-recursion.md) |
| 2026-06-30 | observability | `capped_insert` overgrew the cap intermittently under concurrency (`count == cap + k`) — SurrealDB `kv-mem` snapshot isolation let two concurrent same-key trims each compute `$keep` without seeing the other's just-inserted row, so each under-deleted.
