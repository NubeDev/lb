# Channels scope — rich command/tool responses (the v2 widget contract, un-gridded)

Status: **SHIPPED** — the descriptor-driven contract (the channel is a **pure schema+render interpreter**:
one tool, `tools.catalog`; forms from `input_schema`; responses from `descriptor.result`; a string-keyed
widget/view vocabulary). Kept a scope doc as the authoritative spec of *why*; promotes to `public/channels/`.
Topic: `channels`. Builds on `channels-command-palette-scope.md` (the `/`+`@` request surface),
`channels-query-charts-scope.md` (the first typed `kind:"query_result"` renderer — this generalizes it),
`channels-agent-scope.md` (`agent_result`/`agent_error` — a response this renders), and — load-bearing —
`scope/frontend/dashboard/widget-builder-scope.md` (the **shipped v2 widget contract**: `mount(el, ctx,
bridge)`, the `view` vocabulary, the trust tiers, the grant leash, and the `ext:<id>/<widget>` federation
path). This scope does **not** invent a rendering system; it makes the **channel a second mount surface for
the one already shipped**.

> **Resolved design correction.** An earlier draft of this scope let the **frontend hold tool-specific
> knowledge** — the palette "dispatched `reminder.create`" by reshaping a tool's args client-side and
> "rendered the `reminder.list` response" from a hardcoded render. That is **wrong** and contradicts this
> scope's own premise (rule 7 — the channel is a *generic* front-end for the MCP tool surface). The
> corrected, locked design: **the frontend names exactly one tool, `tools.catalog`, and branches on no
> specific tool name — ever.** For every command it (a) lists it from the catalog, (b) renders its
> `input_schema` through a **string-keyed** widget vocabulary (`x-lb.widget`), and (c) posts the command's
> **descriptor-declared** response render (`descriptor.result`). The descriptor carries **both** halves —
> `input_schema` drives the FORM, `result` drives the RESPONSE — so a new command is *pure backend config*.
> A command with `result` → the palette posts that render (args interpolated into `source.args`); without →
> a plain call. Zero UI change to add a command. The verb accepts the **flat** form fields and does any
> server-side shaping (e.g. `reminder.create` takes schedule+action_kind+fields and builds the nested Action
> itself); the UI posts collected fields **verbatim**. This section supersedes any prose below that reads as
> "the palette assembles the call / hardcodes the response" — the descriptor carries both.

A command, tool, or agent that answers in a channel should be able to return a **rich, typed response** —
a chart, a table, a stat, a form/wizard, a control, or (when generated by an AI) a sandboxed custom UI —
**not just text or a hard-coded `query_result` shape**. The producer declares *how to render* — on the
**`ToolDescriptor` itself** — using the **same `{ view, source|data, options }` cell contract the dashboard
widget builder froze as v2**; the channel renders it with the **same renderer registry, the same trust
tiers, and the same host-mediated bridge** — leashed to the **viewer's** grant, re-checked at the host per
call. The frontend holds **zero tool-specific knowledge**: it names one tool (`tools.catalog`), renders each
command's `input_schema` through a string-keyed widget vocabulary, and posts each command's declared
`descriptor.result` — with **no branch on a specific tool name**. The result: the channel becomes a generic
front-end for the MCP tool surface (rule 7), exactly as the dashboard already is — without a second,
parallel command/UI system to maintain in the client.

## Goals

- **The descriptor carries BOTH sides of the contract.** `ToolDescriptor` gains an optional **`result`**
  field — the `x-lb-render` envelope, the v2 rich-result `{ v, view, source?, options?, action?, tools? }`
  a command's *answer* mounts as. `input_schema` (JSON Schema + `x-lb` vendor hints) drives the **FORM**;
  `result` drives the **RESPONSE**. Both are standard-JSON-Schema-compatible **vendor extensions** — any
  `x-`/vendor key is ignored by an off-the-shelf validator, so the schema stays valid. A command **with**
  `result` → the palette posts that render (the collected args interpolated into `source.args`); **without**
  → a plain call. The producer declares the OUTPUT half exactly where it declares the input half.
- **The palette is a GENERIC interpreter — no tool-name branches.** The UI names **one** tool,
  `tools.catalog`. For every command it: lists it, renders its `input_schema` widgets, and posts its
  `descriptor.result`. It **never** switches on a specific tool name, never reshapes a tool's args, never
  hardcodes a command's response. New response shapes and new commands ship from the backend/an extension
  with **zero** channel-UI changes (the registry resolves an unknown `view`/widget to an honest fallback).
- **The widget/view vocabulary is OPEN: UI built-ins ∪ extension-contributed.** Both the request registry
  (arg widgets keyed by `x-lb.widget`) and the response views resolve a **string** to a renderer: a built-in
  (`cron`/`select`/`table`/`chart`/`switch`/`stat`/…) **OR** an extension-contributed `ext:<id>/<widget>`
  (the shipped `WidgetView`/`ExtWidget` federation path — install-gated, leashed by `[[widget]].scope ∩
  grant`). Unknown → an honest text/summary fallback. So an extension ships a **tool AND the widget its
  form/response renders through**, and the generic palette + `WidgetView` mount it — no core-UI change. This
  is the fullest expression of "reuse the shipped widget contract": dashboard widgets AND extension widgets,
  both resolved by string.
- **Verbs accept the FLAT form; the UI never reshapes.** A command's `input_schema` IS the form the palette
  renders; the **verb** accepts those flat fields and does any server-side shaping (e.g. `reminder.create`
  takes `schedule` + `action_kind` + fields and builds the nested `Action` server-side). The UI posts the
  collected fields **verbatim** — no client-side reshaping of a tool's args (that reshaping was the leak the
  correction removes). The generic `widget:"select"` + `source:"<tool>"` (options from calling another
  catalog tool — the runtime picker is its first tenant) keeps new pickers backend config, not UI PRs.
- **The AI/generative case is one more sandboxed `view`, not a base layer.** When an agent generates a
  novel layout rather than picking a fixed view, it emits **`view:"template"`** (JSX/Plot render template)
  — which already renders in the **sandboxed iframe** tier. A future generative-UI language (A2UI /
  JSON-render / OpenUI-lang) is added as an **additional sandboxed `view` renderer** (e.g. `view:"a2ui"`),
  inheriting the iframe trust wall for free — never a bypass of it.
- **Security identical to a cell.** A rendered response's bridge can call only the tools it declares ∩ the
  **viewer's** install grant, re-checked at the host per call; workspace derived from the viewer's token,
  never the message. No token at the renderer; untrusted/generated code only in the iframe.
- **Ephemeral, not persisted-as-a-cell.** A response lives in the durable channel `Item` body (channel
  history is the record), NOT in a `dashboard:{id}` record. The new surface is the *adapter* from a message
  body to the shipped `mount(el, ctx, bridge)`, not a new persistence layer.

## Non-goals

- **No new render engine, view vocabulary, or trust tier.** This is the widget-builder v2 contract applied
  to the channel; adding views is additive there, not here. If a response needs a view that doesn't exist,
  that's a widget-builder-scope change, consumed here for free.
- **No dashboard builder in the channel.** The channel *renders* producer-declared responses and *drives*
  palette-declared requests; authoring a persisted, grid-placed widget stays the dashboard's job. (A
  "pin this response to a dashboard" is a named follow-up, not this slice.)
- **No un-leashed tools; no widening.** A response can call only tools in its declared set ∩ the viewer's
  grant. Rendering a rich response invents **no** new capability — it *calls* already-gated ones. The AI
  producing a `template` does not grant that template anything the viewer lacks.
- **No in-process untrusted/generated code.** Scripted (`plot`/`d3`/`template`) and any generative-UI
  (`a2ui`/…) responses render in the sandboxed iframe; in-process is trusted-publisher-key-only (unchanged
  from `ui-federation-scope.md`). An AI-authored layout in the shell process is RCE — not done.
- **No AI-only assumption.** A rich response is producer-declared whether the producer is a deterministic
  host verb, an extension, or an agent. The generative (`template`/`a2ui`) path is the AI *option*, not the
  base contract — a plain tool returns `view:"table"`, no model involved.
- **No `*.fake.ts`.** Real gateway, real installed reference tool/extension, real seeded rows, real agent
  run (only the model-provider HTTP may be stubbed via `MockProvider`).

**Skill doc (§6 checklist): N/A for a new one.** This scope adds no new agent-/API-drivable *verb* — it
widens an existing `Item` body shape (the `render` block) and a tool `input_schema` convention (the `x-lb`
enum), both *consumed* by producers whose own surfaces (the palette command surface / `agent.invoke` /
`store.query`) already have their skills. If Phase-1 exposes a new drivable producer verb during
implementation, the session owns writing that verb's `skills/<name>/SKILL.md` — but none is planned here.

## Intent / approach

**A channel rich response is a dashboard cell that isn't on a grid.** The dashboard already ships the hard
part — a `view`→renderer dispatcher (`WidgetView` + `views/*`), a host-mediated `bridge.call/watch`
(`widgetBridge.ts`) leashed by `cell.tools ∩ grant`, an iframe trust router for untrusted/scripted code,
and a versioned (`v:2`) contract. The `channels-query-charts` slice already proved a *single* typed
response (`kind:"query_result"` → an auto-plotted chart) rides channel history + SSE. This scope collapses
those two truths: **generalize the one-off `query_result` renderer into "render the response's `render`
block through `WidgetView`"**, so *any* producer can answer with *any* shipped view — and the AI case is
just the `template`/`a2ui` view on the same seam.

The join, not either half:

- **New (small):** a channel-side **`ResponseView`** — the adapter that reads a `render` block from an
  `Item` body and mounts it via the shipped `WidgetView`, with the bridge scoped to the **viewer's**
  principal and an **ephemeral** cell (data may be **inline** in the body OR a `{tool,args}` source the
  viewer re-runs). Plus the `render` envelope typed in the payload (`payload.types.ts`), and the host/agent
  helpers that emit it.
- **New (small, request side):** promote `x-lb` from two ad-hoc hints (`entity`, `widget:sql`) to a
  **versioned widget enum** with a generic `widget:"select"`/`source:"<tool>"`, and a UI **widget registry**
  keyed by `x-lb.widget` with an **unknown→text fallback** (so a backend can declare a widget an older UI
  doesn't know, and it degrades instead of breaking — the "keep updating the UI" requirement).
- **Reused wholesale (no change):** `WidgetView`, every `views/*` renderer, `widgetBridge.ts`, the iframe
  trust router, the `v` versioning, `render_templates` (a durable generative layout can persist there),
  the palette's arg-rail machinery.

**Why generalize instead of adding response kinds.** The alternative — keep `query_result`, add
`agent_result_chart`, then `tool_result_table`, then `form_response`, then `ai_ui` — multiplies the
payload contract and the channel renderer per response kind (exactly the trap the widget-builder scope
called out for source bindings). One `render: { view, … }` block collapses all of them onto the **one**
render contract the platform already standardizes on. A new response shape = a producer emitting a
different `view`; the channel needs zero changes.

**Why the AI case can't be "just JSON-render everywhere" (your open question, decided).** A deterministic
tool returning tabular data should render through the **fixed, in-process** `table`/`chart` views — fast,
trusted, no code execution. Forcing every response through a generative-UI interpreter would (a) run an
interpreter in the hot path for data that needs none and (b) blur the trust line. So: **fixed views for
declared shapes (in-process, trusted); generative UI (`template`/`a2ui`/JSON-render) only for the
AI-*authored*-layout case, and only in the iframe tier.** The producer picks the tier by picking the
`view`. That is the honest split — the generative languages are *a* view, not *the* substrate.

**Rejected alternatives:**

- *Let the frontend hold per-tool knowledge (reshape a tool's args / hardcode a command's response).*
  Rejected — it forks a **second command system into the client**, breaks "the channel is a generic MCP
  front-end" (rule 7), and means every new command needs a UI change. The descriptor carries the **form AND
  the render** (`input_schema` + `result`); the UI stays a **pure schema+render interpreter** that names
  only `tools.catalog` and branches on no tool name. (This is the design correction this revision records.)
- *A new bespoke channel-response render system.* Rejected — it rebuilds the shipped v2 widget contract
  (renderers, bridge, trust tiers, versioning) in parallel, and an AI can't tell the fork from the truth
  (rule 9's spirit). Reuse the one seam.
- *Make everything generative (A2UI/JSON-render as the base layer).* Rejected — runs an interpreter for
  data that has a fixed view, and puts trusted deterministic output through the untrusted-code tier.
  Generative UI is the AI-authored *option* (`view:"template"|"a2ui"`), sandboxed, not the base.
- *Persist each rich response as a `dashboard:{id}` cell.* Rejected — a channel answer is ephemeral motion
  in history, not an authored, grid-placed, durable widget. It reuses the cell *contract* (shape +
  renderer + bridge), not the cell *persistence*. (Pinning to a dashboard is a follow-up that would create
  a real cell.)
- *Iframe-everything for isolation simplicity.* Rejected — the shipped trust router already renders
  built-in views in-process (safe, they run no author code) and only sandboxes scripted/generated code.
  Iframing a `table` is pure overhead; keep the shipped tiering.

## How it fits the core

- **Tenancy / isolation (rule 6):** the response `Item` is workspace-namespaced (channel history already
  is); `ResponseView`'s bridge derives the workspace from the **viewer's** session token, never the message
  body or an iframe `postMessage`. A ws-B viewer scrolling a channel can render a response only against
  ws-B — a `source` re-run or a `template` write hits the host wall exactly as a cell does. **Mandatory
  two-session test**, including a response whose `source`/`action` names a tool: ws-B cannot execute it
  against ws-A.
- **Capabilities (rule 5/7):** rendering invents **no** new capability. A read view's `source` re-run is
  gated by that tool's existing cap (`mcp:federation.query:call`, `mcp:store.query:call`, …); a control/
  template write by the write tool's cap. The **request** side adds one small read cap for the generic
  picker source it consumes (e.g. `mcp:agent.runtimes:call` for the runtime `select` — see
  `run-lifecycle-scope.md`), list-only. **The deny path is the headline test:** a response whose `render`
  declares a tool the viewer lacks renders the honest "not available" fallback and the host denies the call
  even if the bridge filter were bypassed. Deny is opaque.
- **Placement (rule 1):** `either`. The producer (host verb / agent worker / extension) emits the `render`
  block on whichever node it runs; the channel renders it on whichever node the viewer is attached to. No
  role branch — the response is data in an `Item`, the renderer is the shipped UI.
- **MCP surface (§6.1) — the API shape:**
  - **The descriptor gains `result` — the OUTPUT half of the contract.** `ToolDescriptor` (shipped with
    `{ name, title, group, input_schema }`) gains an optional **`result`** = the `x-lb-render` envelope
    (the v2 `{ v, view, source?, options?, action?, tools? }`). `input_schema` was already the FORM; `result`
    is now the RESPONSE the command's answer mounts as. Additive, `v`-stamped, a standard-JSON-Schema vendor
    extension (`x-`-keyed → ignored by off-the-shelf validators). A command *with* `result` → the palette
    posts that render (collected args interpolated into `source.args`); *without* → a plain call.
  - **The palette is a generic interpreter of the catalog.** It consumes **only** `tools.catalog` (shipped):
    for each command it renders `input_schema` widgets and posts `descriptor.result`. **No branch on a tool
    name.** A rich response is a *shape* on an existing channel `Item` (the worker/agent already posts
    `agent_result`/`query_result`); this scope widens that body with a `render` block **whose contents come
    from the descriptor's `result`**, not a new write verb. `channel.post`/history/stream are unchanged.
  - **Get/list:** none new for responses (channel history IS the list). The **request** side consumes
    `tools.catalog` (shipped) for the command menu and adds the **generic picker read** `widget:"select"`
    calls (an existing read tool **named by the descriptor**, e.g. `agent.runtimes`) — still no tool-name
    branch: the descriptor says which read tool feeds the dropdown.
  - **Live feed:** reused — a response with a streaming `source` (`series.watch`/`bus.watch`) uses the
    shipped SSE via `bridge.watch`; an agent response already streams `RunEvent`s over the run SSE. No new
    transport, no polling.
  - **Batch:** N/A — one response renders one block. A response whose `source` fans a tool over an
    unbounded range MUST name a **job**-backed tool (§6.10), never loop in the renderer (same bound the
    widget-builder scope states).
- **Data (SurrealDB):** the response lives in the channel `Item` body (durable history, state) — no new
  table. **Inline** render data is bounded in the body (reuse the widget-builder inline cap posture,
  `INLINE_MAX_BYTES`); a durable **generative layout** (an AI-authored `template`/`a2ui` doc worth keeping)
  persists in the existing **`render_templates`** table, referenced by id — not a new store. State vs
  motion: the durable answer is the `Item`; the live feed is bus/SSE motion.
- **Bus (Zenoh):** response motion (the item + any `RunEvent`/series stream) is the existing channel/run/
  series SSE — fire-and-forget. A **control/template write** with a must-deliver effect goes through the
  **outbox** via the tool it calls (durability is the *tool's* concern, not the renderer's) — identical to
  the widget-builder control rule.
- **Sync / authority:** the response `Item` is a `(table,id)` upsert on the shipped §6.8 channel sync path;
  a rich response authored on the hub renders idempotently on a reconnecting edge (it re-reads the body and
  re-mounts). A `source`-backed view re-runs its read on render (fresh), an `inline`-data view is
  self-contained.
- **Secrets:** none reach the renderer. A `source`/`action` tool needing a secret pulls it server-side via
  `lb-secrets` inside the tool handler; the secret never touches the `Item` body, the bridge, or the
  iframe.
- **SDK/WIT impact — FLAG.** Three small, versioned boundaries, all additive over the shipped
  `ToolDescriptor`/registry surface and all `v`-stamped: (1) the **`result` field on `ToolDescriptor`** (the
  `x-lb-render` envelope = the v2 `{ v, view, source?, options?, action?, tools? }`) — the OUTPUT half of the
  contract, mirrored in the Rust descriptor and `payload.types.ts`, a standard-JSON-Schema vendor extension
  (`x-`-keyed, so an off-the-shelf validator ignores it and the schema stays valid); (2) the **channel
  `render` envelope** on an `Item` body — additive on the existing `agent_result`/`query_result` items,
  carrying the same `{ v, view, … }` the descriptor's `result` declares; (3) the **`x-lb.widget`
  vocabulary** — an **OPEN, string-keyed** set: UI built-ins ∪ extension-contributed `ext:<id>/<widget>`,
  resolved by string with an `unknown→text/summary` fallback, so an extension shipping a tool can ship the
  widget its form/response renders through. None of the three changes the frozen v2 `mount(el, ctx, bridge)`
  widget contract (this scope *consumes* it unchanged). This is the one boundary a reviewer signs off.

## Example flow

1. **A deterministic tool answers with a table.** Alice runs a palette command whose host verb returns
   `{ render: { v:2, view:"table", source:{ tool:"store.query", args:{ sql:"SELECT …" } } } }` in its
   `Item` body. The channel's `ResponseView` mounts the shipped `table` view; the bridge re-runs
   `store.query` under **Alice's** grant + workspace (from her token) → rows render in history. Trusted,
   in-process, no code executed.
2. **A command with a form/wizard (request side).** Bob types `/` → picks a command whose `input_schema`
   declares `region: {x-lb:{widget:"select", source:"cloud.regions"}}`, `size:{x-lb:{widget:"number"}}`.
   The palette renders a **dropdown** (options fetched by calling `cloud.regions`, gated by its own cap)
   and a number field — a backend-declared form, no channel-UI change. Submit posts the structured call.
3. **The runtime picker is just `widget:"select"`.** The `/agent` command's `runtime` arg is
   `{x-lb:{widget:"select", source:"agent.runtimes"}}` — the same generic dropdown, options from
   `agent.runtimes` (`run-lifecycle-scope.md`), default preselected. No bespoke widget.
4. **An agent answers with a generated UI (the AI case).** Carol asks the agent to "show me a status
   board." The agent emits `{ render: { v:2, view:"template", options:{ code:"<jsx>", …} } }`. It renders
   in a **sandboxed iframe**; its bridge may call only the tools the template declares ∩ **Carol's** grant,
   host-re-checked. A durable board persists as a `render_template:{id}` referenced by the response.
5. **A control in a response.** A monitoring tool returns `{ render:{ view:"switch", action:{ tool:
   "mqtt.publish", argsTemplate:{ topic:"…", payload:"{{value}}" } }, source:{ tool:"mqtt.status" } } }`.
   Toggling it calls the write tool through the bridge, gated + ws-scoped — the shipped control view,
   unchanged.
6. **Unknown view / unknown widget degrades honestly.** A producer emits `view:"sankey"` (not shipped) or
   `x-lb:{widget:"color-picker"}` (older UI). The channel renders the honest fallback (a text/table
   summary; a plain text input) — never a crash. A newer UI ships the renderer; old cells/responses keep
   working (versioned, additive).
7. **Deny.** A response's `template` calls `dashboard.delete` (not in its declared set / not in the
   viewer's grant). The bridge rejects it; the host denies it regardless. Nothing happens; the card shows
   the opaque failure.
8. **Isolation.** Dave (a `mcdonalds` session) scrolls the same channel and renders the same
   `source`-backed response; every bridged call is `mcdonalds` (from his token) — a `kfc` table/topic is
   denied/empty. The wall holds across the rendered response.

## Testing plan

Mandatory categories (`scope/testing/testing-scope.md`) — real gateway, real seeded rows, real installed
reference tool/extension, a real agent run (only the model-provider HTTP stubbed); **no `*.fake.ts`**:

- **Capability deny — per response tool, reads AND writes.** A response whose `render` names a tool
  **outside** the viewer's grant is denied **server-side** (assert the host denies even if the bridge
  filter were bypassed): a read `source` outside grant, a control/template **write** outside grant. Deny is
  opaque; the card shows the honest fallback, not the tool's data.
- **Workspace isolation — two real sessions.** A ws-B viewer rendering a `source`/`action`-backed response
  cannot read/write ws-A; the workspace is taken from the viewer's token, never the `Item` body. A ws-B
  `template` cannot reach ws-A. The two-principal test extended to the rendered-response bridge.
- **Token never crosses the boundary.** The session token appears in **no** in-process bridge arg or iframe
  `postMessage` payload (request, reply, or watch), for read and write responses alike (reuse the
  widget-builder assertion on the channel mount).
- **Trust-tier routing.** A built-in view (`chart`/`table`/…) renders in-process; every
  scripted/generated view (`plot`/`d3`/`template`/`a2ui`) renders **sandboxed**; a non-allow-listed key
  cannot federate a response widget in-process even if it asks. (Reuse the shipped router; assert it on the
  channel surface.)
- **Versioning / graceful degradation.** An unknown `view` and an unknown `x-lb.widget` each degrade to the
  honest fallback (no crash); a `v` newer than the UI understands is rejected/falls back, not mis-rendered.
- **Offline / sync.** The response `Item` replays idempotently on the §6.8 channel sync path; a reconnecting
  edge re-mounts a `source`-backed response (fresh read) and an `inline`-data response (self-contained)
  identically.

Plus this slice's cases:

- **Response round-trip (real gateway):** a host verb / agent returns each response view
  (`table`/`chart`/`stat`/`template`/`switch`) in a channel `Item` → `ResponseView` mounts the shipped
  renderer → the view shows real seeded data / a real control acts. Reuse the dashboard view renderers;
  assert they mount from a channel body.
- **`query_result` migration:** the shipped `channels-query-charts` `kind:"query_result"` response renders
  identically through the new `render`-block path (a v1 `query_result` is a `render:{view:"table"|"chart"}`
  response) — no regression to the shipped chart-in-channel.
- **Request-side widget registry:** the palette renders each `x-lb.widget` (`select`/`sql`/`entity`/
  `text`/`number`), an unknown widget falls back to text, and `widget:"select"`+`source:"<tool>"` fetches
  options by calling that tool (gated) — the runtime picker is the concrete case.
- **Generative-UI (AI) e2e:** an agent emits a `view:"template"` response; it renders sandboxed, its bridge
  is leashed, a declared write succeeds and an undeclared one is denied (host effect asserted).

## Risks & hard problems

- **Producer-declared render is attacker-declared render — the viewer's grant is the only leash.** The
  entire safety story is `response.tools ∩ viewer-grant`, re-checked at the host per call. An AI (or a
  compromised extension) producing a response is **untrusted**; the render block must never widen what the
  *viewer* can do. If the host doesn't re-check a rendered response's call exactly as it re-checks a cell's,
  a channel message becomes a privileged actor. **Load-bearing — the deny test must bite a real ungranted
  call, not a UI message.**
- **Generated/scripted code is RCE if it ever runs in-process.** `template`/`a2ui`/`plot`/`d3` responses
  must iframe; the `sandbox` flags, `event.origin` checks, and CSP must be correct or an AI-authored layout
  isn't sandboxed. Reuse — and re-assert on the channel surface — the shipped router's guards; do not add a
  new in-process path for generated UI.
- **The `render` envelope + `x-lb` enum are forever-costs.** Both are boundaries a producer/extension
  depends on; get the versioned shape right once (a v-stamp + `unknown→fallback`), or a revision is
  expensive. Freeze them here with the `v` field and the fallback rule.
- **Inline render data / template size bloats channel history.** A response embedding large inline data or
  a large layout in the `Item` body bloats history (channel items sync). Bound inline size (reuse the
  widget-builder `INLINE_MAX_BYTES`); push a durable generative layout to `render_templates` by reference.
- **Response ↔ view mismatch / dishonest degradation.** A tool returning a scalar can't be a table; a
  producer picking the wrong `view` must degrade honestly (the shipped renderers already do — a view over
  an incompatible shape shows an honest empty/summary, never a fake value). Assert the honest-degradation
  path.
- **A generative-UI language (A2UI/JSON-render/OpenUI) is a new dependency + attack surface.** Adding one
  as a `view` renderer pulls an interpreter into the iframe tier. Keep it **optional and additive** (a
  registered `view`), behind the same sandbox; do not let it become a required base layer or an in-process
  path. Named as a *follow-up view*, not a Phase-1 blocker.

## Open questions

Decisions taken so the build has no blocking gap; residuals are named follow-ups.

**Resolved (decisions taken):**

- **The frontend holds ZERO tool-specific knowledge.** Decided (the design correction). It names one tool,
  `tools.catalog`; renders each command's `input_schema` widgets; posts each command's `descriptor.result`.
  **No branch on any specific tool name**, no client-side reshaping of a tool's args, no hardcoded response.
- **The render envelope lives on the descriptor as `result`.** Decided. `ToolDescriptor` gains an optional
  `result` = the `x-lb-render` `{ v, view, source?, options?, action?, tools? }`. `input_schema` drives the
  form; `result` drives the response. A command *with* `result` → the palette posts it (args interpolated
  into `source.args`); *without* → a plain call. Additive, `v`-stamped, a valid-JSON-Schema vendor extension.
- **The verb accepts the flat form; the UI never reshapes.** Decided. `input_schema` is the flat form; the
  verb does server-side shaping (nested `Action`, etc.). The UI posts collected fields verbatim.
- **Reuse the v2 widget contract — do NOT build a new render system.** Decided. The channel is a second
  mount surface for the shipped `WidgetView`/`views/*`/`widgetBridge`/trust-router.
- **Response = a `render:{ v, view, source?|data?, options?, action? }` block in the `Item` body** (the v2
  cell shape, ephemeral — history is the record, not a `dashboard:{id}`), sourced from `descriptor.result`.
  Decided.
- **AI/generative UI is the `template` (and future `a2ui`/JSON-render) view, iframe-sandboxed — not the
  base layer.** Decided. Fixed views for declared shapes render in-process/trusted.
- **The `x-lb.widget` / view vocabulary is OPEN — UI built-ins ∪ extension-contributed.** Decided. It is a
  **string-keyed** registry resolving to a built-in (`cron`/`select`/`table`/`chart`/`switch`/…) OR an
  `ext:<id>/<widget>` (the shipped federation path, install-gated, leashed by `[[widget]].scope ∩ grant`),
  with an `unknown→text/summary` fallback. Not a closed enum: a new extension ships a tool AND its widget,
  and the generic palette + `WidgetView` mount it with no core-UI change. The generic `widget:"select"` +
  `source:"<tool>"` (runtime picker) is the first tenant.
- **Security = `response.tools ∩ viewer-grant`, viewer's token for the workspace, host-re-checked per
  call.** Decided (identical to the cell bridge).

**Named follow-ups (not Phase-1 blockers):**

1. **Which generative-UI language for the AI `view`?** — Ship `view:"template"` (the already-supported
   JSX/Plot render template) as the Phase-1 generative surface. Evaluate **A2UI vs JSON-render vs
   OpenUI-lang** as an *additional* `view:"a2ui"` renderer in a follow-up — same iframe tier, additive.
   Proposal: pick one by "smallest safe interpreter that an agent can reliably emit," measured against a
   real agent run; don't add all three.
2. **Pin a channel response to a dashboard.** A "save this response as a cell" that mints a real
   `dashboard:{id}` cell from the response's `render` block (the inverse of un-gridding). Additive; a
   follow-up.
3. **`x-lb` widget enum coverage.** Phase-1 ships `select`/`sql`/`entity`/`text`/`number`/`boolean`/`date`
   + `unknown→text`. `textarea`, `json-editor`, multi-step wizard grouping are additive widgets.
4. **Streaming responses beyond the run feed.** A non-agent response with a long-lived `source` stream
   (`bus.watch`) reuses `bridge.watch`; a richer "live-updating card" contract (partial re-render vs
   re-mount) is a follow-up if a use-case needs it.
5. **Descriptor-drive the legacy palette branches.** The generic result path is `descriptor.result`-driven
   and tool-name-blind. The remaining special-cased palette branches — the legacy `agent.invoke` and
   `federation.query` routes — are still **hardcoded** in the client. Make them **descriptor-declared routes
   like the generic result path** (each command's descriptor carries the render/route, the palette posts it
   with no branch), so the client holds zero per-tool knowledge for those too. Additive; a follow-up.
