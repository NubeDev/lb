# Frontend scope — UI standards (shadcn-first, consistent, responsive)

Status: scope (the ask). Promotes to `public/frontend/frontend.md` once adopted across the shell.

Every surface in `ui/` must look like it came from one app and behave on a phone. We adopt
**shadcn/ui primitives** as the only sanctioned control layer, treat the **Members page +
NavRail sidebar** as the canonical look, and require every view to **reshape for mobile**.
This doc is the standard new code is held to and the migration target for the screens that
predate it. The "why": shadcn was intended from day one but never got wired in — the shell
grew its own parallel control layer (`.control-field`, `.soft-button`, `.page-header`, …
in `globals.css`), so today only `features/members/MembersView.tsx` and
`features/shell/NavRail.tsx` are 100% on the primitives. That fork is the breakdown; this
scope closes it.

## Goals

- **shadcn-first.** Every button, input, select, badge, dialog, table, etc. is a component
  from `ui/src/components/ui/*` (generated by the shadcn CLI), not a raw `<button>`/`<input>`
  styled inline and not a bespoke `globals.css` class.
- **One consistent look.** Every full-screen surface follows the Members template: a
  `<section className="flex h-full min-w-0 flex-col">` wrapper with `<AppPageHeader>` at the
  top, the same spacing rhythm, the same tokens (`bg`, `fg`, `muted`, `accent`, `border`).
- **Responsive by default.** Every view resizes cleanly from desktop down to a phone — no
  fixed pixel/`w-64`-style widths that don't collapse, multi-column layouts stack, and the
  sidebar uses its mobile off-canvas sheet.
- A short, enforceable checklist a session (human or AI) can run before calling a UI change done.

## Non-goals

- A visual redesign. Tokens, density, amber accent, and the "operator console" mood are
  unchanged (`ui-design-scope.md`, `frontend-scope.md`). This is about *which primitives and
  layout rules* we use, not a new palette.
- A big-bang rewrite of all 37 legacy views in one PR. Migration is incremental (see
  "Migration"), but **no new screen may add to the legacy pattern.**
- Net-new product surface. No new pages, just the convention the pages obey.

## Intent / approach

shadcn/ui is "copy the component into the repo and own it," which fits FILE-LAYOUT (one
component per file, ours to edit) and the existing Tailwind-token setup. The sidebar already
proves it: `components/ui/sidebar.tsx` is the upstream shadcn sidebar, wired to our tokens,
and it brings real mobile behavior for free (off-canvas `Sheet` under `md`, an `useIsMobile`
hook, icon-collapse). Members proves the page side: `Button` + `Input` + `AppPageHeader`
(which composes `Badge`), nothing hand-rolled.

The standard is therefore: **generate the missing primitives with the shadcn CLI, point them
at our tokens, and migrate views onto them** — deleting the parallel `globals.css` control
classes as their last caller goes. The rejected alternative is keeping the bespoke
`.control-field`/`.soft-button` layer "because it already looks right": it looks right but
it's a second, undocumented component system an agent can't tell from the real one, it has no
variants/a11y baked in, and it's exactly the fork that produced today's inconsistency. One
control layer, owned in `components/ui/`, is the call.

### Canonical reference (copy this shape)

`features/members/MembersView.tsx` — full-screen surface:

```tsx
<section className="flex h-full min-w-0 flex-col bg-bg text-fg">
  <AppPageHeader icon={Users} title="Members" description="…" workspace={ws} actions={…} />
  {/* scrollable body: min-h-0 flex-1 overflow-y-auto */}
  {/* footer form: Button + Input from @/components/ui/* */}
</section>
```

`features/shell/NavRail.tsx` — navigation, on the full `Sidebar` suite (`collapsible="icon"`,
tooltips when collapsed, mobile sheet via `SidebarProvider`/`useIsMobile`).

## How it fits the core

- **One responsibility per file (FILE-LAYOUT §4):** each primitive is its own file in
  `components/ui/`; each view/sub-view stays one component per file. The standard reinforces
  this, it doesn't bend it.
- **MCP / data:** N/A — this is the presentation layer only. It changes no MCP tools, records,
  or bus subjects.
- **No mocks / no fake backend (CLAUDE §9):** N/A to the styling, but the standard's tests run
  through the existing `*.gateway.test.tsx` harness against a real seeded node — a restyle must
  keep its gateway test green, never swap in a fake to "make the snapshot pass."
- **Tenancy / capabilities:** unchanged — `AppPageHeader` surfaces the workspace; cap-gating of
  surfaces stays in `NavRail`'s `allowed`/`extSlots` props and the gateway re-check.
- **SDK/WIT impact:** none — the shell's component layer is not the plugin boundary. (Extension
  UIs federated via `ext-host` should follow the same look, but that's their bundle.)

## The standard (the enforceable checklist)

A UI change is done only when:

1. **Controls are shadcn.** No raw `<button>`/`<input>`/`<select>`/`<textarea>` and no
   `globals.css` control class (`control-field*`, `soft-button*`, `danger-button-sm`,
   `icon-button`, `scope-pill`, `state-alert`, `page-header*`). Use the `components/ui/*`
   primitive; if it doesn't exist yet, generate it (see "Component backlog") rather than
   hand-rolling.
2. **Header is `AppPageHeader`.** Full-screen surfaces wrap in
   `<section className="flex h-full min-w-0 flex-col">` and lead with `<AppPageHeader>` — not a
   custom `<header>` and not the `.page-header` class.
3. **It resizes.** Verified at a narrow viewport (≤640px): no horizontal scroll, fixed side
   columns collapse or move into the mobile sheet, multi-column splits stack
   (`flex-col md:flex-row`, not a bare `w-1/2`), and no un-overridden fixed width (`w-64`,
   `w-96`, `w-[34rem]`) on a top-level region. Use the `sm:`/`md:`/`lg:` prefixes and
   `useIsMobile` where a layout genuinely differs.
4. **Tokens, not raw colors.** `bg`/`fg`/`muted`/`accent`/`border`/`destructive` — no
   `text-red-600`/`bg-zinc-…` literals (the legacy `danger-button-sm` baked these in; the
   `destructive` Button variant replaces it).
5. **Tests stay green** through the real gateway harness; a11y labels (`aria-label`,
   `aria-current`) are preserved as Members/NavRail set them.

## Component backlog (generate before first use)

Present: `button`, `input`, `badge`, `sheet`, `sidebar`, `tooltip`. Missing primitives the
legacy views imply — generate with the shadcn CLI and bind to our tokens as each migration
needs it:

- **`select`** — replaces ~15 raw `<select>` (schema builder, widget picker, visibility).
- **`dialog`** — replaces the hand-rolled `role="dialog"` in `ConfirmDestructive` and
  `CreateSeriesWizard` (identical backdrop/panel/animation duplicated).
- **`table`** — replaces raw `<table>` in the admin views and `DataView`.
- **`switch`** — replaces `SchemaForm`'s hand-rolled `role="switch"` toggle.
- **`alert`** — replaces the `.state-alert` error-banner class.
- **`tabs`** — replaces the raw `<button>` tab rows in `AdminView`/`DataView`.
- **`card`** — for the dashboard widget chrome and dense panels (optional; widgets may keep
  their own chrome if it already reads as a card).

`Button` already covers `.soft-button*` (default/secondary) and `.danger-button-sm`
(`variant="destructive"`); `Input` covers `.control-field*`; `Badge` covers `.scope-pill`;
`AppPageHeader` covers `.page-header*`. No new primitive needed for those — just migrate.

## Migration

Incremental, highest-traffic first; the gate is "no new debt," not "convert everything now":

1. **Freeze the legacy layer.** New/edited views use shadcn only — now **mechanically
   enforced** by ESLint (`ui/eslint.config.js`, run via `pnpm lint`): the standard is an
   *error* everywhere, downgraded to a *warning* only for the files still in that config's
   `LEGACY_VIEWS` allowlist. A new violation in any already-conforming file fails the lint.
2. **Per area, swap in place:** Members is done. Then `channel/` (composer + list),
   `admin/*` (tables + the `w-1/2` splits → `md:` stacks), `dashboard/*`, `ingest/*`,
   `data/*`, `inbox`/`outbox`/`workflow`, `agent`, `extensions`, `session/LoginView`.
   Each swap keeps the view's `*.gateway.test.tsx` green.
3. **Delete on last caller.** When a `globals.css` control class has no callers, remove it.
   The file shrinks as the primitives take over — that deletion is the proof the fork closed.

The mechanical loop: migrate a view → **remove its path from `LEGACY_VIEWS`** in
`ui/eslint.config.js` (it now errors on regression) → re-run `pnpm lint`. When `LEGACY_VIEWS`
is empty, flip the `lint` script to `eslint src --max-warnings 0` so the standard is fully
build-breaking. Today: `pnpm lint` exits 0 with 141 warnings across 25 legacy views.

## Testing plan

From `scope/testing/testing-scope.md`, the categories that apply to a presentation-layer
standard:

- **Gateway component tests (real node, no fakes).** Each migrated view keeps/extends its
  `*.gateway.test.tsx` — behavior (post a message, add a member, approve an item) must pass
  unchanged after the restyle. This is the "no mocks" guarantee for UI.
- **Responsive assertion.** Add a lightweight check (render at a narrow container / assert the
  mobile branch) for at least the shell and one representative view, so "resizes on mobile"
  is regression-guarded, not eyeballed once.
- **Capability-deny / workspace-isolation:** inherited, not re-proven here — a restyle must not
  remove the cap-gating `NavRail` already encodes (`allowed`, `extSlots`); the gateway
  re-checks regardless.
- **a11y smoke:** the `aria-label`/`aria-current`/roles Members and NavRail set are retained.

## Risks & hard problems

- **Two systems during migration.** Until the legacy classes are gone, both patterns exist —
  an agent could copy the wrong one. Mitigation: this doc + the checklist + "freeze legacy."
- **Token drift in generated components.** shadcn CLI output ships with its own `--background`
  etc.; each generated primitive must be repointed at our `bg`/`fg`/`accent`/`border` tokens
  (as `sidebar.tsx` already is) or it'll look subtly off.
- **Responsive regressions in dense tables.** The admin/data tables are the hardest to make
  mobile-friendly; a `Table` primitive plus horizontal-scroll-in-a-card may be the pragmatic
  answer rather than full reflow.
- **Mobile interaction gaps.** Off-canvas sheet, tap targets ≥ the 8/7/12 sidebar sizes, and
  focus management in `Dialog` need real narrow-viewport testing, not just CSS.

## Open questions

- **Do we run `shadcn add` against the live registry, or keep generating by hand** matching
  `sidebar.tsx`'s already-token-bound style? (Lean: use the CLI, then repoint tokens — less
  drift than hand-authoring.)
- **Is `card` a primitive or do dashboard widgets keep bespoke chrome?** Decide when migrating
  `dashboard/widgets/`.
- **Mobile nav entry point.** On a phone the `Sidebar` is an off-canvas sheet — where does its
  trigger live in each surface's header? (Likely a `SidebarTrigger` slot in `AppPageHeader`.)
- **Lint enforcement.** ✅ Resolved — `ui/eslint.config.js` bans the legacy `globals.css`
  control classes and raw `<button>`/`<input>`/`<select>`/`<textarea>`, forward-only via the
  `LEGACY_VIEWS` allowlist (plus `react-hooks` recommended rules). Remaining sub-question:
  wire `pnpm lint` into CI / the gateway-test job so the freeze runs on every change.
