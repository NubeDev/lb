## Gap 1 — relay reactors are never booted (BLOCKER) — ✅ CLOSED (RouterTarget + BootConfig.outbox_providers + node/tests/relay_boot_test.rs)

`spawn_relay_reactors` (`rust/crates/host/src/outbox/relay_reactor.rs:30`) has **zero
production call sites**. Node boot (`rust/node/src/builder.rs` → `boot_full` →
`rust/node/src/reactors.rs::spawn`, ~L13-43) spawns flow/agent/approval/insight reactors
but never the relay; no `EmailTarget`/`PushTarget` is constructed anywhere at boot. So
`invite.create` and `notify.send` enqueue outbox effects that nothing drains in a
running node — the tests pass because they call `relay_outbox` directly.

**Fix:** register `EmailTarget` and `PushTarget` with `spawn_relay_reactors` in
`rust/node/src/reactors.rs::spawn` (same pattern as the other reactors). Providers come
through the existing `EmailProvider`/`PushProvider` traits — boot needs a provider
injection seam (BootConfig or equivalent additive knob) so a product host can hand in a
real provider; default to a no-op/logging provider when none is configured (never crash
boot). Add a **boot-level test**: boot a node (or the closest harness), mint an invite,
assert the recording email provider actually received the send via the *spawned* reactor
— i.e. prove drain-at-boot, not just loop-body. Same for one push.

Known-deferred (do NOT block the tag): real WebPush VAPID provider, FCM/APNs, SMTP —
fakes/no-op are fine; the traits are the seam.

## Gap 2 — i18n: the catalog engine exists but is connected to nothing — ✅ CLOSED (all four surfaces; see the session doc)

lb's multi-lang is **real but confined to `rust/crates/prefs`**: en+es MF1 catalogs
(`rust/crates/prefs/src/catalog/builtin/{en,es}.mf`), parser/plural/interpolate/fallback
(`src/catalog/`), a settable member `language` axis (`src/prefs.rs:25`,
`src/resolve.rs:49-55`, enabled langs `["en","es"]` in `src/axis/language.rs:10`), a
generated byte-identical client twin (`ui/src/lib/prefs/catalog.generated.ts`, generator
`src/bin/gen_catalog.rs`, renderer `ui/src/lib/prefs/renderMessage.ts`), and a **CI
key-parity gate** (`rust/crates/prefs/tests/generated_catalog_test.rs:62-75`, runs in
`cargo test --workspace`). Use this engine for everything below — do not invent a second
mechanism.

Per-surface verdicts and fixes (all generic; catalogs hold words, records hold keys):

### (a) Locale on member + invite (pre-auth) — PARTIAL
- Member `language` pref: exists, settable. ✅
- **Invite record has NO locale** (`rust/crates/authz/src/invite.rs:34-65`);
  `invite_create` takes none (`rust/crates/host/src/invites/create.rs:26-49`); accept
  seeds no pref. **Fix:** additive serde-default `locale: Option<String>` on `Invite`,
  a param on `invite.create` (validated against the enabled-language axis), carried into
  the outbox effect payload, and on accept **copy it into the new member's `language`
  pref**. Pre-auth read: the invite-accept page must be able to render in the invite's
  locale (expose it on the token-verify response, or equivalent generic seam).

### (b) Invite email — MISSING
Hardcoded English subject/body (`rust/crates/host/src/outbox/email_target.rs:93-99`);
`EmailMeta` has no locale. The old "no templating in core" non-goal is **overturned** by
the multi-lang requirement — note the reversal in the invites scope doc. **Fix:** render
subject/body through the prefs catalog engine (new catalog keys in both `en.mf` and
`es.mf`) using the effect's locale, fallback `en`. Test: same invite, `es` recipient →
Spanish subject+body asserted on `RecordingEmailProvider`.

### (c) Push — MISSING
`notify.send` stores the sender's literal title/body in the effect
(`rust/crates/host/src/notify/verbs.rs:72-101`); `PushTarget` sends identical strings to
every device and reads only `push_muted` (`push_target.rs:124-126`), never `language`.
**Fix:** let `notify.send` accept a catalog **key + args** (keep literal title/body as a
compat path — literals are never translated); at deliver time `PushTarget` resolves each
recipient's `language` pref (it already fetches their prefs) and renders per-recipient.
Test: one notify, two members with `en`/`es` prefs → two differently-localized payloads
on `RecordingPushProvider`.

### (d) Shell + extension-UI catalog seam — PARTIAL/MISSING
- `@nube/minimal-shell` (`packages/minimal-shell/src/`): **zero i18n** — login,
  invite-accept, errors are hardcoded. **Fix:** all user-facing strings through a
  catalog, en+es complete; locale resolution = user pref → `navigator.language` →
  `en` (the browser-language middle link exists nowhere today — add it).
- `@nube/ext-ui-sdk` (`../lb-ext-ui-sdk`): **no catalog API**. **Fix (additive):** expose
  a register-your-catalog + translate seam so extension UIs ship their own en+es
  catalogs and resolve locale the same way. Remember: **its `dist/` is committed —
  rebuild before committing**, and tag it if it changes.
- Note `ui/src/lib/prefs/renderMessage.ts` currently has no non-test callers — wiring
  the shell/SDK makes it live instead of dead code.
- Completeness gate: the `.mf` key-parity test covers server catalogs; ship an
  equivalent check for the shell/SDK TS catalogs (a test or script CI can run) so a key
  added in one language fails the build.
