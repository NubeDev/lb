# i18n — en + es on every user-facing surface (shipped 2026-07-11, `node-v0.2.0`)

One catalog mechanism end to end: the prefs **MF1 catalog engine** (`rust/crates/prefs` —
`en.mf`/`es.mf`, parser/plural/interpolate, CI key-parity gate, generated client twin) is the
single source of words; records and code hold **keys**.

## Server-generated content

- **Invite emails** — `invite.create {locale}` (validated against the enabled-language axis)
  stores the locale on the invite and its email effect; `EmailTarget` renders
  `invite.email.subject`/`invite.email.body` through the catalog in that locale (en fallback).
- **Pre-auth accept page** — `GET /public/invite/verify?workspace=…&token=…` returns
  `{email, locale, redeemable}` (token-gated + rate-limited) so the page renders in the invitee's
  language before any session exists. On accept the locale is copied into the member's
  `language` pref.
- **Push** — `notify.send {title_key, body_key, args}` carries a catalog reference; `PushTarget`
  renders **per recipient** in each recipient's `language` pref at deliver time. Literal
  `title`/`body` remain the compat path (never translated).

## Shell + extension UIs

`@nube/ext-ui-sdk` ≥ 0.7.0 ships the catalog seam: `resolveLocale` (user pref →
`navigator.language` base → `en`), `makeTranslator` (locale → en → key-literal, `{arg}`
interpolation, never blank), `catalogParity` (the CI key-parity gate — a key added in one language
fails the build). `@nube/minimal-shell` ≥ 0.2.0 routes every string through en+es catalogs and
gates parity in its vitest suite. Rich MF1 (plural/select/typed formats) stays server-side via the
generated client twin — the SDK seam is deliberately the simple-interpolation subset.

Adding a language = drop a catalog file (`.mf` + TS map) and enable it on the language axis — no
code change (the mechanism never assumes two languages).
