# Deploy

Deploying a Lazybones `node` to Fly.io as a single-container, Caddy-fronted standalone
node — login, MCP, SSE, agents, flows, rules, insights, all in one binary — with an
embedded SurrealDB store on a persistent volume and the federation sidecar wired for
SQLite datasources. No hosted or bundled Postgres (the node embeds its one datastore).

Full scope + design rationale: [`docs/scope/deploy/fly-deploy-scope.md`](../../../../docs/scope/deploy/fly-deploy-scope.md).
Implementation session: [`docs/sessions/deploy/fly-deploy-implementation-session.md`](../../../../docs/sessions/deploy/fly-deploy-implementation-session.md).

## One image, three drivers

The build inputs — `Dockerfile`, `Caddyfile`, `entrypoint.sh`, local `compose.yml` — live
once in [`deploy/common/`](../../../../deploy/common/) and are reused **verbatim** by every
driver: a local Docker build, GitHub CI's `deploy-image` job, and `fly deploy`. The
repo-root `.dockerignore` is the one ignore file all three actually read (Docker resolves
`.dockerignore` from the build context root, not next to the Dockerfile — see the deploy
debugging entries if you're tempted to add a per-Dockerfile one).

```
                    https://<app>.fly.dev/
                              │  fly-proxy (TLS)
                    ┌─────────▼──────────────┐
                    │ Caddy  :8080           │
                    │  /, /assets/*, /shims/*│ → SPA bundle (embedded in image)
                    │  everything else       │ → 127.0.0.1:8731
                    └─────────┬──────────────┘
                    ┌─────────▼──────────────┐
                    │ node (gateway)         │
                    │  127.0.0.1:8731        │
                    │  store → /data/store   │──┐
                    └─────────┬──────────────┘  │  one Fly volume `lb_data`
                    ┌─────────▼──────────────┐  │      mounted at /data
                    │ federation sidecar     │  │
                    │  sqlite: /data/demo/*  │◄─┘
                    └────────────────────────┘
```

The SPA is a browser build with `VITE_GATEWAY_URL=""` (same-origin) — every API call is a
relative request Caddy proxies to the gateway, so a new backend route never needs a
Caddyfile change.

## Try it locally

```bash
make fly-local        # build deploy/common/Dockerfile, run behind mkcert TLS on https://localhost
make fly-smoke         # POST /login, GET /workspaces, datasource.list — real HTTP against a real node
make fly-local-down    # tear down (keeps volumes)
```

`smoke.sh` proves the whole chain end to end against a real, running container: the SPA is
served, login mints a working token, the gateway is reachable *through* Caddy (not the SPA
fallback), and the demo `demo-buildings` SQLite datasource — generated by
`docker/postgres/seed.py --sqlite` and registered via the real `datasource.add` verb — is
queryable. No mocks anywhere in this path.

## Deploy to Fly

```bash
fly apps create lazybones
fly volumes create lb_data --region cdg --size 1
fly secrets set LB_SIGNING_KEY="$(openssl rand -hex 32)"   # stable sessions across restarts
make fly-deploy
```

`LB_SIGNING_KEY` matters: without it, the node mints a fresh Ed25519 signing key every
boot (fine for `make dev`) and every session dies on the next restart. Set it once as a
Fly secret and every login survives a redeploy — verified locally by minting a token,
restarting the container, and confirming the same token still authenticates.

Full first-time setup, secrets, and operator commands: [`deploy/fly/README.md`](../../../../deploy/fly/README.md).
