## Series plane: paging, decimation, retention (shipped 2026-07-14)

The native series plane (`series.read` over the ingest-committed `series` table) is now
production-shaped (issues #55–#58, one PR):

- **Schema + time semantics** — `ts` is a real SurrealDB `datetime` (idempotent migration for legacy
  numeric rows); named indexes `series_seq_idx (series, seq)` and `series_ts_idx (series, ts)`;
  commit converts wire `labels` to tag edges once per series (so `series.find` discovers ingested
  series); a per-workspace **series cardinality cap** (default 10 000) dead-letters samples naming a
  new series past the cap. `series.read` accepts wall-clock `{from, to}` (epoch ms, half-open)
  alongside `from_seq`/`to_seq`.
- **Keyset paging** (`mode:"rows"`, the default) — `{limit, cursor, direction}` in,
  `{samples, next_cursor, prev_cursor}` out. Seeks the unique `(seq, producer)` composite (tie-safe);
  the cursor is an opaque versioned bookmark, re-authorized against `mcp:series.read:call` on every
  page and useless under another workspace's token. Default/max `limit`: 10 000.
- **Bucketed decimation** (`mode:"buckets"`) — `{from, to, width_ms | budget}` in,
  `{buckets: [{t, min, max, avg, last, count}], width_ms}` out (≤ budget, hard cap 2 000; sparse).
  Spikes survive in `min`/`max`; `last` is exact. Computed server-side as a chunked fold over the
  indexed window — a month of 1s samples renders as ~1 000 buckets, never millions of rows.
- **Retention + rollup tiers + GC** — per-prefix policies
  (`{prefix, raw_for_ms, tiers: [{width_ms, keep_for_ms}]}`) via the admin-capped
  `series.retention.set / list / delete / gc` verbs. GC rolls raw history into stored tiers
  (sum+count carried → exact re-aggregation), then evicts; bucketed reads transparently merge the
  finest tier where raw was evicted. The series table no longer grows forever.

## Query-result cache (shipped 2026-07-20)

Dashboards re-run the same SQL on every load, every refresh tick, and for every viewer. The
federation child now keeps a **TTL-bounded cache of query results**, so a repeat of an identical
query inside a caller-declared freshness window is served from memory instead of paying another
round-trip to the source. It completes the three-layer stack: no connect cost (pool cache), no
transport queue (call concurrency), no repeat query.

**It is off unless you ask for it.** Add a freshness window to `federation.query`:

```json
{ "source": "warehouse", "sql": "SELECT …", "cache": { "ttl_s": 30 } }
```

- **No `cache` field** → today's behaviour, bit for bit. `ttl_s` is **required** inside `cache` (an
  empty `cache: {}` is a schema error) — the child never invents a freshness contract for you.
- `ttl_s: 0` disables caching for that call.
- **Node kill-switch:** `LB_FEDERATION_RESULT_CACHE=off` on the node process forces every call to
  bypass regardless of what callers ask for. The child inherits it at spawn; restart to apply.

**Who sets `ttl_s`:** the surface that knows its own refresh contract — a dashboard page refreshing
every 30 s tolerates ~30 s of staleness. Nothing in lb core decides a TTL.

**The staleness bound, stated honestly.** With `ttl_s` equal to your refresh interval, a tick can
land on a 29.9 s-old entry, hit, and serve it for another full cycle: worst case is
**`ttl_s` + your refresh interval**, not `ttl_s`. Set `ttl_s` slightly below the interval (≈0.9×) if
the tighter bound matters. An external writer mutating the remote database directly is bounded
**only** by `ttl_s` — that is the contract.

**What invalidates an entry:** `federation.write` and an applied `federation.migrate` drop every
cached result for that source, and so does a failed `datasource.test` probe. Writes made *outside*
lb are not seen — only the TTL bounds those.

**Bounds and footprint.** Per child: 128 entries, 64 MB total, 4 MB per entry (a larger result is
served but not cached). The cache is process-local and children are per `(workspace, extension)`, so
a cross-workspace hit is impossible by construction — and a node's worst-case footprint is **64 MB ×
active workspaces**. A multi-tenant cloud node may want that cap lowered. The cache is lost on
restart, harmlessly: the next query is simply slow.

**Observability.** Each query event on the child's stderr carries `result_cache: hit|miss|bypass`,
plus `age_ms` on a hit — which is what lets a UI badge "data as of Xs ago". Events never carry the
DSN or raw SQL (SQL appears only as `sql_digest`).

## Per-target query diagnostics: why a panel is blank (shipped 2026-07-21)

`viz.query` resolves each of a panel's targets by re-entering the dispatcher under the caller's
authority. It used to **collapse any target failure to an empty frame and throw the reason away**
(`Err(_) => Vec::new()`), so four very different outcomes — a capability deny, a bad query, an
operational error, and a query that genuinely matched zero rows — all reached the client as the same
`{fields:[],length:0}` frame with HTTP 200. A blank tile could never say **why** it was blank.

Each frame now carries an additive **`status`** — one of `ok`, `empty`, `denied`, `error`:

```jsonc
{ "refId": "A", "fields": [], "length": 0,
  "status": { "state": "error",
              "message": "parse error: … No field named nonexistent_col …" } }
```

- **`ok`** — ran, ≥1 row. `message` absent.
- **`empty`** — ran, 0 rows. `message` absent (the UI writes its own "0 rows for &lt;range&gt;").
- **`error`** — the downstream tool's own message: the bad SQL you typed, the malformed arg you
  passed. It is *your own request echoed back to you*, so surfacing it leaks nothing — and withholding
  it was the actual papercut.
- **`denied`** — a target you lack the capability for. **`message` is always absent** — the deny is
  opaque by construction: it never reveals a gate, a tool's existence, or cross-workspace shape, and
  its status is byte-for-byte identical whether or not the named data exists (no enumeration oracle).

**Additive and backward-compatible.** Same verb, same `mcp:viz.query:call` cap — no new verb, no new
capability. `status` is `#[serde(default)]`, so a legacy client that ignores it renders exactly as
before, and the compute-only (frames-in) path — which resolves no target — carries no `status` at all.
A failed target still yields an empty frame; no rows are ever fabricated. The consuming
empty-state/status-bar UI is downstream (rubix-ai `dashboard-first-paint-scope.md`).

## Panel resolution negotiation: the dashboard asks for buckets (shipped 2026-07-24)

The [series plane's decimation](#series-plane-paging-decimation-retention-shipped-2026-07-14) shipped
`series.read {mode:"buckets", from, to, width_ms}` — an O(buckets) spike-safe read — but nothing asked
for it: every chart fetched **raw rows** and `viz.query` **dropped** the `maxDataPoints`/`minInterval`
that ride the panel. This is the negotiation seam that turns a panel's **visible time range + point
budget** into a bucket width and passes it down — natively for SurrealDB series, by pushdown for
federated tiles. One contract, both engines, chosen per target by where the data lives.

**The derivation (`host/src/viz/resolution.rs`), Grafana's model, not a fixed ladder:**
`width = ceil(range / budget)`, snapped **UP** to a human step ladder
(`1s 5s 10s 30s 1m 5m 10m 15m 30m 1h 2h 3h 6h 12h 1d 7d 30d`), floored by `minInterval`, clamped to
`MAX_BUCKETS` (2000). The resolution scales with the range for free — the point payload stays ~flat at
every window:

| Range | budget 1000 → width | points |
|---|---|---|
| 6 h | 30 s | ~720 |
| 1 month | 1 h | ~720 |
| 1 year | 12 h | ~730 |
| 2 years | 1 d | ~730 |

**Series targets, native.** A mode-less `series.read` chart target with a numeric window is upgraded
in place to `{mode:"buckets", width_ms}` — the shipped decimation path, rollup-tier merge included. An
explicit `mode:"rows"` (tables/exports/the raw inspector) or an explicit `width_ms` (aligned
multi-series overlay) **always wins**. The width is a pure function of `(range, budget, minInterval)`,
so N viewers of a panel derive byte-identical args → the quantised cache keys collapse them to one
compute.

**Federated targets, by pushdown — interval macros (zero SQL parsing).** `viz.query` substitutes four
Grafana-compatible macros into a `federation.query` target's SQL before dispatch, with the SAME derived
resolution:

```sql
SELECT date_bin(INTERVAL '$__interval', ts, TIMESTAMP '1970-01-01') AS bucket, avg(v), max(v), min(v)
FROM readings WHERE ts BETWEEN $__timeFrom AND $__timeTo GROUP BY bucket
```

- `$__interval_ms` → the width in ms (integer) · `$__interval` → the same as a Grafana duration string
  (`"12h"`) · `$__timeFrom`/`$__timeTo` → the window (epoch ms). So a tile coarsens itself as the range
  grows. **An un-macro'd SQL is byte-identical** — a hand-SQL tile runs verbatim. (The structured
  `decimate` arg with per-dialect `time_bucket`/`date_bin`/`strftime` and a guaranteed spike-safe record
  is the deferred v1.5.)

**Additive, capability- and workspace-safe.** No new verb, no new capability: `viz.query` starts
honoring fields it already accepted. An injected `mode:"buckets"` rides the same `mcp:series.read:call`
grant; a macro-substituted SQL the same `federation.query` gate. A denied target is still an honest
empty frame; a ws-B panel resolves nothing from a ws-A series regardless of mode. The **dashboard sends
the inputs** (rubix-ai): the fetch hook threads the window (zoom → refetch at the auto-derived width),
`$__interval` becomes a computed built-in, and line/area renderers draw a min–max envelope so a spike
that survived decimation survives on screen. Against an un-bumped lb pin the extra args degrade to
today's raw fetch (feature-detected by the frame shape, not the version).
